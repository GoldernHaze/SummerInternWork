## **STEP 1: Serialization — "why do we add 5 extra bytes?"**

```c
ngham_payload[p++] = (uint8_t)(chunk.occultation_id >> 8);   // 0x03
ngham_payload[p++] = (uint8_t)(chunk.occultation_id & 0xFF); // 0xE8
ngham_payload[p++] = chunk.sequence_number;                  // 0x01
ngham_payload[p++] = chunk.total_chunks;                     // 0x04
ngham_payload[p++] = chunk.data_size;                        // 0x20
memcpy(&ngham_payload[p], chunk.data, chunk.data_size);       // the 32 data bytes
```

**Why serialize at all?** Once this packet is transmitted and received on the ground, the receiver only sees raw bytes — it has no idea "this belongs to occultation 1000, chunk 1 of 4" unless we **embed that metadata inside the NGHam payload itself**. That's the whole point of Lucas's requirement: *"assign a sequence number, ID of the occultation... data size"* — these fields must travel **inside** the packet, not just exist in our RAM.

**Real result:**
```

payload (hex)  = 03 E8 01 04 20 | 10 11 12 13 ... 2F
                 └─────┬─────┘   └──────┬───────┘
              metadata (5 bytes)   actual data (32 bytes)
```

Note: `1000` in decimal = `0x03E8` in hex — that's why you see `03 E8` at the start. This confirms occultation_id round-trips correctly.

## **STEP 3: Header Byte (Padding Size)**

```c
padding_size = k - 3 - payload_len;   // k=63 (from RS(79,63))
header = padding_size & 0x1F;
```

**Real calculation:**
```
padding_size = 63 - 3 - 37 = 23
header = 23 = 0x17
```

Why "`- 3`"? Because `k` (63) must hold: `1 header byte + payload + 2 CRC bytes + padding`. So `padding = k - 1 - payload_len - 2 = k - 3 - payload_len`.

## **STEP 4: CRC + Full Data Block**

```c
uint16_t crc = ngham_crc16(data_block, pos); // over header+payload only (38 bytes)
```

**Real result:**
```
CRC16-CCITT = 0x56EC
```

This is calculated over `[header(1) + payload(37)] = 38 bytes`. Notice it does **NOT** include padding — this matches the spec exactly (CRC is computed before padding is added, so the receiver can strip padding and still verify CRC against just the real content).

**Full data_block (63 bytes = `k`):**


Total: `1 + 37 + 2 + 23 = 63` ✓ exactly matches `k`.

## **STEP 5: Reed-Solomon Parity (the hard part)**

This is where the real "magic" happens. The RS encoder treats those 63 data bytes as one giant polynomial and mathematically divides it by a special "generator polynomial" (built from Galois Field arithmetic) — the **remainder** of that division becomes the 16 parity bytes.

**Real result:**
```
parity (16 bytes) = FC 48 46 EB C6 99 CC 1D 63 10 FA 05 65 74 E2 8B
```

**What these bytes actually let you do:** if, during radio transmission, up to **8 of those 79 total bytes** get corrupted by noise (any 8 bytes, anywhere in the 63+16 block), the receiver can use these 16 parity bytes to **mathematically reconstruct the exact original 63 bytes** — without needing to ask for a retransmission. That's the entire value of NGHam over a simpler protocol.

*(Honest caveat: I used the standard "textbook" GF(256) field polynomial `0x11D` for this demo — the real NGHam spec uses `0x187` with a specific generator root. The **mechanism** and **math structure** here are fully correct and real, but for byte-exact compatibility with SpaceLab's actual ground station decoder, these specific constants would need to be matched exactly — worth flagging to Lucas once we get this far.)*
